# Topic 1.2 — Calculus (for AI/ML)

> **Why this topic exists on your roadmap:** Every time a model "learns," it's doing one thing: adjusting its internal numbers a tiny bit to make its errors smaller. Calculus is the tool that tells you *which direction* and *how much* to adjust each number. Without it, "training a model" is just a magic incantation. With it, you can look at a loss curve going down and know exactly what's happening under the hood.

**Prerequisite:** Topic 1.1 (Linear Algebra) basics — you'll be applying derivatives to vectors and matrices, not just single numbers.

## 1. Derivatives

### The idea
A derivative tells you the **rate of change** of a function at a specific point — if you nudge the input a tiny bit, how much does the output move, and in which direction? Geometrically, it's the slope of the line that just barely touches the curve at that point (the tangent line).

### Relatable analogy
Think of your car's speedometer. Your position (miles driven) is a function of time. Your speed at any instant is the *derivative* of position with respect to time — literally "how fast is this thing changing right now." If your speed is positive, you're moving forward; if it's zero, you've stopped; if it's negative, you're reversing.

### Where it shows up in ML
- The **loss function** (how wrong your model is) is a curve. The derivative of that curve tells you whether increasing a specific weight will make the error go up or down, and by roughly how much.
- This is the single idea that "training" is built on.

### Exercises
1. **By hand:** For `f(x) = x²`, the derivative is `f'(x) = 2x`. Compute the slope at `x = 1`, `x = 3`, and `x = -2`. What does the sign tell you about which way the curve is heading at each point?
2. **By hand:** For `f(x) = 3x + 5` (a straight line), what is the derivative everywhere? Why does it make sense that it's a constant?
3. **NumPy — numerically estimate a derivative (no symbolic math, just definition):**
   ```python
   import numpy as np

   def f(x):
       return x**2

   def numerical_derivative(f, x, h=1e-5):
       return (f(x + h) - f(x)) / h   # slope between two very-close points

   for x in [1, 3, -2]:
       print(f"f'({x}) ≈ {numerical_derivative(f, x):.4f}  (true value: {2*x})")
   ```
4. **Reflect:** Why do we use a *tiny* `h` (like 0.00001) instead of something like `h = 1`? What would go wrong with a large `h` on a curvy function?

## 2. Partial Derivatives

### The idea
Real ML models don't have just one input — they have thousands or millions of weights. A **partial derivative** asks: "if I wiggle *just this one* input/weight, and freeze everything else, how does the output change?" You do this separately for every single weight.

### Relatable analogy
Imagine you're baking a cake and the result (taste score) depends on both oven temperature and baking time. A partial derivative with respect to temperature asks: "if I only change the temperature a little, keeping the time exactly the same, how much does the taste score change?" You'd ask the same question independently for time. Each ingredient/knob gets its own "sensitivity report."

### Where it shows up in ML
- A neural network might have millions of weights. Training needs the partial derivative of the loss with respect to *each individual weight*, so it knows how to adjust every single one.
- The collection of all partial derivatives (one per parameter) is called the **gradient** — covered next.

### Exercises
1. **By hand:** For `f(x, y) = x²y + 3y`, compute:
   - `∂f/∂x` (treat y as a constant)
   - `∂f/∂y` (treat x as a constant)
2. **By hand:** Evaluate both partial derivatives at `x=2, y=1`.
3. **NumPy — numerical partial derivatives:**
   ```python
   def f(x, y):
       return x**2 * y + 3*y

   def partial_x(f, x, y, h=1e-5):
       return (f(x + h, y) - f(x, y)) / h

   def partial_y(f, x, y, h=1e-5):
       return (f(x, y + h) - f(x, y)) / h

   print("∂f/∂x at (2,1):", partial_x(f, 2, 1))
   print("∂f/∂y at (2,1):", partial_y(f, 2, 1))
   ```

## 3. The Gradient

### The idea
The **gradient** is just all the partial derivatives of a function, bundled together into one vector. It points in the direction of **steepest increase** of the function. Flip its sign, and you get the direction of steepest *decrease* — exactly what you want when trying to minimize error.

### Relatable analogy
You're standing on a foggy hillside and want to get to the bottom (lowest error) but can't see far. The gradient is like feeling the ground under your feet — it tells you which direction is "uphill" right now. To go downhill, you simply step in the *opposite* direction of the gradient, then re-check your footing (recompute the gradient) and repeat.

### Where it shows up in ML
- This hillside analogy **is** gradient descent, the core training algorithm behind nearly every ML/DL model.
- The gradient vector has one entry per model parameter — for a network with a million weights, the gradient is a million-dimensional arrow pointing "downhill."

### Exercises
1. **By hand:** For `f(x, y) = x² + y²` (a bowl shape), write the gradient vector `∇f = [∂f/∂x, ∂f/∂y]`.
2. **By hand:** Evaluate the gradient at the point `(3, 4)`. Which direction should you move to *decrease* `f` the fastest?
3. **NumPy — visualize gradient descent on a bowl:**
   ```python
   import numpy as np

   def f(x, y):
       return x**2 + y**2

   def grad(x, y):
       return np.array([2*x, 2*y])

   point = np.array([3.0, 4.0])
   lr = 0.1  # learning rate — how big a step to take

   for step in range(10):
       g = grad(point[0], point[1])
       point = point - lr * g   # move opposite to the gradient
       print(f"Step {step}: point={point}, f={f(point[0], point[1]):.4f}")
   ```
   Watch `f` shrink toward 0 as you "walk downhill."

## 4. The Chain Rule

### The idea
When one function is nested inside another — `f(g(x))` — the chain rule lets you find the derivative of the whole thing by multiplying the derivative of the outer function by the derivative of the inner function. It's how you differentiate "a function of a function."

### Relatable analogy
Think of a factory assembly line: raw material → Machine A shapes it → Machine B paints it → final product. If you want to know "how much does the final product change if I tweak the raw material slightly," you multiply the sensitivity of each machine in the chain: (how much B's output changes per unit change in A's output) × (how much A's output changes per unit change in raw material). Multiply the sensitivities down the line, and you get the end-to-end sensitivity.

### Where it shows up in ML
- **This is backpropagation.** A neural network is literally a chain of functions (layer 1 → layer 2 → layer 3 → loss). To know how a weight in layer 1 affects the final loss, you multiply the derivatives layer by layer, all the way from the loss back to that weight. That's the entire backpropagation algorithm — repeated chain rule, nothing more mysterious than that.

### Exercises
1. **By hand:** Let `f(x) = (3x + 1)²`. Treat this as outer function `h(u) = u²` where `u = g(x) = 3x + 1`. Compute `dh/du`, then `du/dx`, then multiply them to get `df/dx`. Double check by expanding `(3x+1)²` directly and differentiating — do you get the same answer?
2. **By hand:** For a 2-step chain `y = sin(x²)`, apply the chain rule: what's `dy/dx`?
3. **NumPy — verify numerically:**
   ```python
   import numpy as np

   def f(x):
       return (3*x + 1)**2

   def numerical_derivative(f, x, h=1e-5):
       return (f(x + h) - f(x)) / h

   x = 2.0
   analytical = 2 * (3*x + 1) * 3   # chain rule result: 2u * 3
   numerical = numerical_derivative(f, x)
   print("Analytical:", analytical, " Numerical:", numerical)
   ```
4. **Mini backprop by hand:** Suppose `loss = (prediction - actual)²`, and `prediction = w * x` (a tiny 1-weight linear model). Given `x=2, w=3, actual=10`:
   - Compute `prediction`.
   - Compute `loss`.
   - Use the chain rule to compute `d(loss)/d(w)` (hint: `d(loss)/d(prediction)` first, then `d(prediction)/d(w)`, then multiply).
   - This single number is exactly what a training step would use to update `w`.

## 5. Gradient Descent (bringing it together)

### The idea
Gradient descent is the training loop: compute the gradient of the loss with respect to every parameter, take a small step in the opposite direction, and repeat until the loss stops improving.

### Relatable analogy
Continuing the foggy-hillside picture: you feel the slope under your feet (gradient), take one careful step downhill (update the weights), then feel the slope again from your new spot, and repeat. The size of each step is the **learning rate** — too large and you might stumble past the valley floor and overshoot; too small and it takes forever to get down.

### Where it shows up in ML
- This is *the* training algorithm for linear regression, logistic regression, and every neural network you'll build in later phases (with variants like SGD, Momentum, Adam layered on top in Phase 1.4 and Phase 5).

### Exercises
1. **From-scratch mini-project — fit a line without scikit-learn:**
   ```python
   import numpy as np

   # Fake data: y = 2x + 1, plus a little noise
   np.random.seed(0)
   X = np.linspace(0, 10, 50)
   y_true = 2 * X + 1
   y = y_true + np.random.normal(0, 1, size=X.shape)

   # Start with random guesses for slope (w) and intercept (b)
   w, b = 0.0, 0.0
   lr = 0.01

   for epoch in range(1000):
       y_pred = w * X + b
       error = y_pred - y

       # Partial derivatives of Mean Squared Error w.r.t. w and b
       dw = (2 / len(X)) * np.sum(error * X)
       db = (2 / len(X)) * np.sum(error)

       # Gradient descent update
       w -= lr * dw
       b -= lr * db

       if epoch % 100 == 0:
           loss = np.mean(error ** 2)
           print(f"Epoch {epoch}: w={w:.3f}, b={b:.3f}, loss={loss:.3f}")

   print(f"\nFinal: w={w:.3f}, b={b:.3f} (true values were w=2, b=1)")
   ```
2. **Experiment:** Re-run with `lr = 0.5` and then `lr = 0.0001`. What happens in each case? (One should overshoot/diverge, the other should crawl too slowly.) Note what you observe.
3. **Extend:** Plot the loss value at every epoch using `matplotlib`. This "loss curve going down" is exactly what you'll see in every training log for the rest of this roadmap — now you know it's just gradient descent doing its thing.

## 6. Jacobians & Hessians (conceptual — good to know, not urgent to master)

### The idea
- **Jacobian**: when a function outputs *multiple* values (not just one), the Jacobian is the full table of all partial derivatives — every output with respect to every input. It generalizes the gradient to multi-output functions.
- **Hessian**: the "derivative of the derivative" — a matrix of second-order partial derivatives. It tells you about the *curvature* of the loss surface (is it a gentle bowl or a sharp narrow valley?).

### Relatable analogy
If the gradient tells you which way the hill slopes, the Hessian tells you the *shape* of the hill — is it a smooth wide bowl (easy to descend) or a steep narrow canyon (easy to overshoot and bounce off the walls)? Some advanced optimizers use this shape information to take smarter steps.

### Where it shows up in ML
- Jacobians show up in backpropagation once a layer has multiple outputs (which is every real layer).
- Hessians show up conceptually when discussing why some loss surfaces are hard to optimize, and in a few advanced optimization methods (Newton's method, natural gradient) — you won't need to compute these by hand day-to-day; deep learning frameworks (PyTorch/TensorFlow, coming in Phase 5) do it via **automatic differentiation**.

### Exercises
1. **Conceptual only:** For `f(x, y) = [x + y, x * y]` (2 outputs, 2 inputs), how many partial derivatives make up the full Jacobian? (Hint: outputs × inputs.)
2. **Reflect:** Why might a very "curvy" (high-curvature) loss surface make training less stable than a smooth bowl-shaped one, even with the same gradient direction?
